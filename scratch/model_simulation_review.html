Model Generation & Simulation Review Notes

1. build_hazards Does Everything (src/modelgeneration.jl)

  • Current state: single function parses formulas, injects intercepts, names parameters, generates runtime functions, allocates storage, and assembles ParameterHandling views.
  • Risk: Seven+ responsibilities blur intent and make incremental changes (e.g., spline support or shared-baseline changes) hard to reason about.
  • Suggestion: extract helpers per family (_build_markov_hazard, _build_weibull_hazard, …) plus a shared “schema → design matrix” helper. Smaller units would isolate tricky logic and make testing simpler.

2. “Mega Constructor” multistatemodel (src/modelgeneration.jl)

  • Current state: ~200 lines that validate data, enumerate hazards, build transition matrices, emission matrices, surrogate hazards, and choose concrete model types.
  • Risk: Deep branching hides invariants and complicates error reporting; future additions (e.g., new censoring modes) require touching the monolith.
  • Suggestion: split into helpers like _validate_inputs!, _build_transition_structures, _make_emissions, _select_model_type. Each block becomes easier to document and unit-test.

3. Nested Control Flow in simulate_path (src/simulation.jl)

  • Current state: one loop interleaves survival inversion, Tang cache toggling, RNG draws, optimizer fallback, transition sampling, and censoring bookkeeping.
  • Risk: Hard to see when u should reset or when covariate caches refresh; debugging new features (e.g., hazard-specific RNG tweaks) requires mentally tracing the entire loop.
  • Suggestion: extract helpers such as _event_occurs_in_interval, _solve_jump_time!, _advance_interval!, _choose_next_state!. Flatter structure clarifies the state machine and exposes reusable pieces.

4. Hard-Coded Truncation Logic in Diagnostics (test/diagnostics/generate_model_diagnostics.jl)

  • Current state: truncated_distribution_functions normalizes cdf_base(horizon) and assumes a single upper bound.
  • Risk: Logic is tightly coupled to “event before horizon” scenarios inside diagnostics; if we need truncated comparisons elsewhere (likelihood tests, documentation notebooks), we’ll likely copy the same code.
  • Suggestion: consider moving truncation into a general utility that accepts arbitrary [lower, upper] bounds so other components can reuse it without duplicating assumptions.

5. DIP/OCP Pressure in build_hazards (src/modelgeneration.jl)

  • Current state: hazard family selection is controlled by string comparisons ("exp", "wei", "gom", "sp") with copy-pasted blocks that perform identical scaffolding steps.
  • Risk: Every new baseline type forces edits to this function and couples high-level orchestration to concrete hazard structs, undermining extension via multiple dispatch.
  • Suggestion: introduce lightweight family-specific builder types (e.g., ExponentialSpec) and dispatching helpers so future hazard families register themselves instead of modifying the central factory.

6. Boolean-Driven Composition in Simulation APIs (src/simulation.jl)

  • Current state: simulate relies on data, paths, and time_transform booleans to toggle large blocks of logic, often allocating containers that never get returned.
  • Risk: Flag explosion creates dead branches, nested conditionals, and unclear invariants (e.g., paths == false && data == false defense). It also blocks composition with alternative strategies.
  • Suggestion: expose explicit entry points (simulate_data, simulate_paths) and thread the time-transform behaviour through a strategy object so behaviour changes come from composition rather than configuration flags.

7. Monolithic Jump Solver in simulate_path (src/simulation.jl)

  • Current state: the per-subject sampler interleaves RNG draws, survival evaluation, root finding, transition selection, covariate refresh, and censoring bookkeeping inside one loop.
  • Risk: Difficult to test individual concerns (e.g., bisection accuracy) or to reason about when caches reset; any optimisation touches the entire routine.
  • Suggestion: split into helpers like _locate_jump_time, _advance_interval!, and _update_covariates!, letting simulate_path orchestrate the state machine rather than implement every detail.

Follow-up Actions to Consider

  • Refactor ownership boundaries: Introduce per-family builders plus dedicated validation/emission helpers so model generation mirrors ColPrac expectations (small, single-purpose functions).
  • Strengthen testability: After extracting helpers, add targeted unit tests (e.g., check that _build_transition_structures rejects duplicate states) to catch regressions earlier than integration tests.
  • Document truncation utility: When generalizing the truncated CDF/PDF helper, document expected bounds and edge cases (zero mass, horizon = ∞) so downstream likelihood or diagnostic code can rely on the same math.
